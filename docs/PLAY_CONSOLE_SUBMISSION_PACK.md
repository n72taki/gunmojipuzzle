## Closed Test Package

Before closed testing:

1. Generate Android wrapper from resolved TWA settings.
2. Publish app shell and policies on the final HTTPS host.
3. Publish `/.well-known/assetlinks.json` with the final release certificate fingerprint.
4. Replace local policy links with hosted URLs.
5. Run `tools/release-gate.cjs`.
6. Build an isolated hosted-policy public preview with `PUBLIC_PREVIEW_OUTPUT_DIR=dist/public-preview-hosted-check`, then run `tools/check-public-preview.cjs --expect-hosted-policies` and `tools/check-production-readiness.cjs --root gunmojipuzzle\dist\public-preview-hosted-check --skip-native`.
7. Run `tools/check-build-metadata.cjs` and archive the exact build id with the closed-test notes.
8. Run `tools/check-production-readiness.cjs --expect-blockers` and confirm remaining blockers are closed-test-intentional, not accidental public-release blockers.
9. Use `docs/CLOSED_TEST_PLAN.md` for tester recruitment, 14-day script, stop conditions, and production access notes.

## Production Submission Blockers

These must be resolved before public production submission:

- Final HTTPS public host.
- Final hosted privacy policy, terms, and commercial transactions URLs.
- TWA host, manifest URL, icon URLs, package id review, and release signing fingerprint.
- Digital Asset Links deployed to the production host.
- Native Android build/version/signing pipeline.
- Android target API level aligned with the current Play requirement.
- Decision to remove monetization or connect real ad/billing SDKs correctly.
- Server authority for stamina, paid recovery, rankings, or any competitive/economy feature.
- Actual Data safety form matching the shipped SDKs and data flows.
- Final target audience/content rating choices matching the shipped build.
- Device QA on low-end Android, modern Android, small iPhone, and large iPhone.

## Four-Perspective Submission Review

- Player: store listing must show the board, 60-second loop, stamina/practice distinction, daily free gift, and replay reasons without test-only economy confusion.
- Educator: listing and app content should describe word notes as light learning hooks, not formal curriculum or guaranteed educational outcomes.
- Streamer: feature graphic, screenshots, and description should show word-call, result sharing, and sound checks without implying unauthorized music or third-party assets.
- Viewer: store assets should be understandable at small sizes and show the completed word/readable reward moment.
- Release: all submission answers must match the exact build uploaded to Play Console, not the local prototype or a future intended build.
